## Authentication Flow Updates

The OTP authentication flow works as follows:

1. **Phone Entry:** User enters a 10-digit Indian mobile number in the input field next to the +91 country code box. The "Send OTP" button remains disabled until exactly 10 digits are entered.

2. **Send OTP (mock):** `mockAuthService.sendOTP()` is called with a 1-second simulated delay. It succeeds for any valid 10-digit number, returning `{ success: true, message: "OTP sent successfully" }`.

3. **OTP Entry:** The screen transitions to the OTP state with a fade-in animation. The subtitle updates to show a masked version of the phone number. A 30-second resend countdown begins.

4. **Verify OTP (mock):** `mockAuthService.verifyOTP()` is called with a 1.5-second delay. Test credentials: phone `9876543210` + OTP `123456`.
   - **Success:** Returns `{ success: true, data: { token: "mock_token_abc123", userId: "mock_user_001" } }` → navigates to `RegistrationNextStep`
   - **Failure:** Returns `{ success: false, message: "Incorrect OTP. Please try again." }` → inline red error banner appears

5. **Post-verification:** RegistrationNextStep → Continue → PersonalInfoScreen → OnboardingNext

## Architecture Improvements

### Mock Service Layer Pattern
The `mockAuthService.js` file acts as a drop-in replacement for a real API service. It simulates network latency (1s for send, 1.5s for verify) so the UI behaves realistically during development. The screen imports `sendOTP` and `verifyOTP` by name — when the real backend is ready, only the import path changes (or the mock file internals are replaced). No screen code needs modification.

### Service → Screen Separation
All API logic lives in service files (`mockAuthService.js`, `OtpService.js`). Screens only call exported functions and react to the response shape. This keeps screens focused on UI and state management.

### Backend Connection Path
The `OtpService.js` file contains commented-out `fetch()` implementations showing exactly how to connect to real endpoints. The backend developer can:
1. Implement the stubs in `backend/controllers/`
2. Uncomment the fetch calls in the service file (or replace the mock)
3. No screen changes required

### Return Shape Contract
All service functions follow a strict contract:
- **Success:** `{ success: true, data: { ... } }`
- **Failure:** `{ success: false, message: "Human readable error" }`

This ensures screens can handle responses uniformly regardless of whether the backend is mock or real.
